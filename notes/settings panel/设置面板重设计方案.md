# 设置面板重设计方案

日期：2026-05-17

## 1. 目标

在**保留所有现有设置、按钮、下载区域、动态插槽**的前提下，重新组织设置面板里的元素布局，让它同时解决下面几个问题：

1. 设置太多，扫一眼时难以区分。
2. 当前 3 个选项卡的信息架构不稳定，同一类设置被拆散到了不同地方。
3. 下载进度区域和下载按钮被压在第二个选项卡里，开始下载后不够醒目。
4. 置顶功能虽然实用，但会进一步打散原有分类。
5. 当前面板更像“把所有设置排成一列”，效率高，但辨识度、层次感、可维护性都偏弱。

我的结论是：**最值得采用的方向，不是继续微调现在的 3 个横向选项卡，而是改成“左侧导航 + 右侧工作区”的结构。**

---

## 2. 已确认的约束

这次重设计不应该破坏下面这些既有能力：

- 保留所有现有设置项，不删功能。
- 保留现有按钮区、下载区、进度条区。
- 保留动态插槽，因为最终 HTML 是由多个模块拼出来的。
- 保留“置顶选项”机制。
- 保留不同页面类型下的差异显示逻辑。
- 保留移动端适配能力。

换句话说，这次改的重点不是业务逻辑，而是**信息架构、布局层级、视觉组织方式**。

---

## 3. 我对现状的判断

### 3.1 目前的问题不只是“选项太多”

真正的问题其实有 4 个层面：

| 层面     | 当前情况                                     | 带来的问题                                         |
| -------- | -------------------------------------------- | -------------------------------------------------- |
| 一级结构 | 现在是 Crawl / Download / More 3 个选项卡    | More 承担了过多职责，像一个“杂物间”                |
| 二级结构 | More 里再分 5 个分类                         | 一部分同类设置又被拆到前两个选项卡，分类逻辑不稳定 |
| 行级结构 | 每个 option 基本都只是横向一行               | 当行内控件一多，就会变成一长串文字和开关           |
| 状态结构 | 置顶、高级设置、下载中、新功能等状态是叠加的 | 用户能操作，但不容易形成清晰心智模型               |

### 3.2 当前界面最核心的优点不能丢

虽然现在不够美观，但它有一个非常强的优点：**高频操作离结果很近**。

例如：

- 抓取相关按钮就在抓取设置附近。
- 下载相关设置和下载进度在同一个大区域。
- 大多数设置都能直接改，不需要经过太多层跳转。

所以重设计不能走成“更漂亮，但操作路径更长”。  
我的建议是：**保留效率优先，但把效率建立在更清晰的结构上，而不是建立在把所有东西直接摊开上。**

### 3.3 旧的“显示高级设置”机制不必保留

旧方案里之所以需要“显示高级设置”，本质上是因为：

- 设置项堆在一个较长的列表里。
- 用户很容易被过多设置淹没。

但在新的结构里，已经有：

- 一级导航
- 二级分组
- 首页（仪表盘）
- 置顶设置

这些机制已经足够承担“减轻认知负担”的职责。  
因此，后续重构时不必继续保留“显示高级设置 / 简洁模式”这一套逻辑，更合适的替代方案是：**保留全部设置，只控制所有二级分组的展开 / 折叠状态。**

---

## 4. 重设计时应该坚持的原则

### 4.1 先解决“找不到”，再解决“好不好看”

优先级应该是：

1. 让用户更快找到设置。
2. 让用户更容易理解某个设置属于哪一类。
3. 让下载状态更容易被看到。
4. 最后再做视觉美化。

### 4.2 一级分类要稳定，二级分组要贴近任务

一级分类不应该再出现“这类设置的一部分在 A，一部分在 B”的情况。  
一级分类一旦确定，就尽量稳定。

二级分组则应该更贴近用户任务，例如：

- “筛选抓取范围”
- “图片过滤”
- “命名规则”
- “保存格式”
- “下载行为”

而不是只按代码模块或历史演化痕迹分组。

---

## 5. 新的方案：标题栏 + 左侧导航 + 右侧工作区

这是我最推荐的方案。  
它比现在改动大一些，但收益也最大，而且最能同时兼顾：

- 结构清晰
- 高频操作效率
- 下载状态可见性
- 后续继续加设置时的可扩展性

### 5.1 总体框架

建议把整个面板改成下面这种结构：

```text
┌──────────────────────────────────────────────────────────────┐
│ 标题栏：logo、名称 | 搜索框、展开/折叠全部、赞助、关闭           │
├──────────────┬───────────────────────────────────────────────┤
│  首页        | 首页：置顶设置 + 抓取按钮区 + 附加功能区 + 下载区 │
│  抓取        │ 抓取分类内容区                                  │
│  命名        │ 命名分类内容区                                  │
│  下载        │ 下载分类内容区                                  │
│  增强        │ 增强分类内容区                                  │
│  通用        │ 其他分类内容区                                  │
│  帮助        │ 帮助按钮、超链接                                │
│  搜索        │ 搜索结果区域（用户搜索时才会显示）               |
├──────────────│───────────────────────────────────────────────┤
│  下载摘要栏   │ 显示总进度、暂停下载、停止下载等简要状态         │
└──────────────────────────────────────────────────────────────┘
```

#### 标题栏的内容和布局

##### 内容

标题栏的内容分为 2 个大板块。

板块 1: 
- logo（文件是 `\src\static\icon\logo128.png`）
- 名字
- 只在移动端页面里显示的关闭按钮。它总是显示在这个板块的最右侧。 

板块 2:
- 第 1 个区域放置搜索框和“展开/折叠全部”。
- 第 2 个区域放置不重要的按钮，目前里面只有赞助按钮。点击赞助按钮的效果等于点击目前下载器底部的“赞助我”，会显示一个消息框来显示提示。这个区域只在 PC 端页面里显示，在移动端页面里隐藏，以减少占据的空间。
- 第 3 个区域放置只在 PC 端页面里显示的关闭按钮。

##### PC 端的显示样式

在 PC 端，标题栏的两个板块是左右两端布局的。即板块 1 位于左侧，板块 2 位于右侧。

##### 移动端的显示样式

在移动端页面里，标题栏里的两个板块是从上到下显示的，每个板块占据 1 行。每行的内容都是居中显示的。

#### 主体区域的结构和内容

在 PC 端页面里，主体区域是左右布局的。

由于添加了左侧区域，所以设置面板需要适当加宽，这样右侧显示按钮、设置的区域依然可以使用与之前一致或接近的宽度，不会因为宽度变小而产生更多的换行。

左侧区域的总宽度为 180 px，左右两侧有 10px 的 padding。

右侧区域的总宽度为 675px，左侧有 10px 的 padding，右侧则是 15px 的 padding（因为右侧可能会有滚动条，所以 padding 比左侧大一些）。

主体区域有最大高度限制（继承现在的样式即可）。

当高度不足时，左右两个板块都可以上下滚动。因为板块是分开的，所以它们的滚动需要是独立的，一侧板块滚动时，另一侧不能跟着滚动。

##### 左侧的内容

###### 一级分类导航

主体区域是左右结构。左侧从上到下显示一级分类的选项卡，分类名字依次是：
- 首页
- 抓取
- 命名
- 下载
- 增强
- 通用
- 帮助
- 搜索

备注：
- 点击一级分类就会激活它，并在右侧显示该分类里的内容。
- 默认激活的分类是“首页”，它的作用是仪表盘。
- 同时只能激活一个一级分类。
- 需要记录上一次激活的分类、当前激活的分类，在有需要时使用。当搜索页面隐藏之后（因为用户清除了搜索内容），需要自动切换到上一次激活的分类。至于当前激活的分类，可能会在一些代码逻辑里使用。
- 每当激活左侧的一级分类时，隐藏其他分类的右侧区域，并显示当前分类的右侧区域。还需要检查它里面应该显示哪些设置项。如果某个设置项应该显示在这里但却没有显示（因为它可能被移动到了搜索结果里），就把这个设置项移动过来。
- 被置顶的设置只会显示在“首页”和搜索结果里，不会显示在其他一级分类里。在其他分类里不需要显示该分类里被置顶的设置。
- “搜索”分类平时是隐藏的，所以用户不能主动激活它。只有当需要显示搜索结果时才会激活搜索分类，并在右侧区域显示搜索结果。

###### 下载摘要栏

在左侧区域的底端，停靠着一个下载摘要栏。它是默认隐藏的，只有当首次触发了抓取完成事件 `crawlComplete` 之后，才会显示。

下载摘要栏里会显示两排内容：
- 第一排显示下载状态和下载总进度，例如 `➡️ 6/83`。下载状态使用 emoji 符号表示，根据当前下载状态的不同，会显示对应的符号：未开始下载时是 ▶️，下载中是 ➡️，暂停下载时是 ⏸️，停止下载时是 ⏹️，下载完成后是 ✅。
- 第二排显示 3 个控制按钮，依次是：开始下载 ▶️，暂停下载 ⏸️，停止下载 ⏹️。

为了让用户可以确认每个按钮的功能，因此每个按钮都应该有 title 属性，这样当用户把鼠标放到对应按钮上时可以查看提示。

##### 右侧区域的内容

- “首页”的右侧内容区域：有 4 个子区域。第一个区域是被置顶的设置，内容是一个标题卡片（名字为置顶的设置）+ 每个被置顶的设置项的卡片。接下来的区域依次是抓取按钮、附加功能、下载区域，这 3 个区域都使用“标题+内容卡片”。置顶的设置、抓取按钮区域默认展开；附加功能区域和下载区域默认折叠。有下载任务时自动展开下载区域。
- 中间 5 个设置分类的右侧内容区域：有多个二级分类的设置，每种分类都是标题卡片+内容卡片的组合。
- “帮助”分类的右侧内容区域：它里面是多个小按钮或超链接，每个按钮都使用内容卡片，并且帮助分类里的卡片都是只有部分宽度的，一行可以显示多个。

备注：
由于右侧区域里的二级分类和按钮区域都可以切换展开、折叠状态，因此需要记录每个可折叠区域的折叠状态。

二级分类标题需要支持黏性滚动：
- 当右侧区域滚动时，如果某个二级分类区域被滚动遮挡了一部分内容（即上半部分在显示区域之外），那么该二级分类的标题卡片会始终显示在分类区域的顶部，不会被滚动到显示区域之外。
- 保持只有一个标题卡片是处于黏性滚动的。如果有多个分类标题卡片符合黏性滚动的条件（即有多个二级分类区域被滚动到显示区域之外），只显示它们里的最后一个（因为这个标题就是当前正在查看的分类的标题）。

##### 移动端的显示样式

在移动端页面里，把主体区域的内容改成**上下布局**：

- 一级导航改成横向排列。如果一行放不下就会自动换行。
- 分类内容区域显示在下方。

这样可以避免在窄屏上左右两列都过窄，导致大量换行。

#### 搜索功能

##### 搜索框的样式

在左侧显示一个搜索图标（不可点击，因为不需要点击它）。当输入文字之后，在右侧显示一个 x 图标，点击它可以清除输入的搜索词。示意图如下：
![搜索框示意图](./搜索框示意图.png)

##### 搜索匹配范围

- 从所有设置项的配置数据里查找（而不是从 DOM 元素里查找）。不查找“帮助”里的内容，因为它里面的内容不属于设置。
- 查找每个设置项的名字。需要注意的是，设置名字有多语言版本，只从与当前界面语言相同的名字里查找，不要查找其他语言版本的名字。备注：设置名字的字符串里可能含有 html 标签（通常是 span 标签），所以需要在设置的数据里添加一个属性来保存过滤了 html 标签后的名字。
- 查找每个设置的关键字。这是因为有些设置的名字里含有一些同义词，用户可能输入了同义词，此时如果关键字里有同义词的话就可以搜索到这个设置了。

##### 如何呈现搜索结果

- 用户输入搜索词时，自动查找并显示搜索结果。为了避免短期内多次计算的压力，需要使用防抖处理（等用户停止输入 0.2 秒之后才开始查找搜索结果）。
- 不在搜索框下面显示下拉的候选项列表，因为这样会增加复杂度。而且目标设置本身就可以在设置面板里显示，所以没太大必要显示下拉候选项。
- 当用户在搜索框里输入内容之后（搜索词非空），就激活“搜索”分类，并在其右侧内容区域（全局搜索页面）里显示符合条件的设置。
- 搜索页面的顶部会显示有多少条搜索结果。如果没有符合条件的搜索结果，显示“没有找到符合条件的设置。你可以尝试搜索更简短的关键词。”。如果有搜索结果，则显示“找到 x 条设置”。
- 把搜索到的设置项从它们的二级分类区域里移动到搜索结果区域里。这些设置项可以直接使用，所以点击它们时，不需要跳转到它们原本所属的一级分类里。
- 在搜索到的设置里，如果设置的标题里有命中的文字，则高亮显示命中的文字。
- 搜索结果可能会来自多个一级分类，在呈现搜索结果时，这些设置的显示顺序应该与左侧一级导航一致：依次是：首页 → 抓取 → 命名 → 下载 → 增强 → 其他。
- 在每个一级分类内部，再按原本的二级分组顺序显示。也就是以“一级分类”+“二级分类”为搜索结果进行分组，例如 `下载 / 文件格式`，并使用标题卡片来显示这个组合的分类名称。

何时隐藏搜索页面：
未搜索或搜索词为空时，隐藏搜索分类页面，并重新激活上一次激活的一级分类。搜索词为空的情况可能由多种触发方式：
- 当用户点击搜索框右侧的 x 图标时会清除搜索词
- 当用户手动清空了搜索词时（例如连续按 backspace 键）。

可以通过监控搜索框 input 控件的 value 来检测搜索词是否为空。

### 5.2 一级导航改成 7 类

把现在 More 里的 5 个大类真正提升为一级导航，这样结构会稳定很多。另外我还添加了 2 个额外的一级导航：“首页”和“帮助”。

| 一级导航 | 主要承载内容                                                                                                 | 说明                                                                                 |
| -------- | ------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------ |
| 首页     | 置顶的设置、抓取按钮区域、附加功能区域、下载区域                                                             | 该分类里的内容相当于仪表盘首页。该分类会默认显示                                     |
| 抓取     | 抓取范围、过滤条件、抓取策略、抓取按钮                                                                       | 对应“决定抓什么”                                                                     |
| 命名     | 文件名规则、文件夹规则、序号、标签别名                                                                       | 对应“决定文件叫什么”                                                                 |
| 下载     | 保存格式、下载线程、下载策略、下载触发                                                                       | 对应“决定如何保存和下载”                                                             |
| 增强     | 缩略图、预览、复制、高亮、界面增强                                                                           | 对应“增强使用过程”                                                                   |
| 通用     | 外观、语言、日志、导入导出、设置管理                                                                         | 系统级、全局级选项                                                                   |
| 帮助     | 显示原本位于下载器底部的帮助按钮和链接 |

### 5.3 样式方面的一些设想

参考 Clash Verge 的配色方案（截图文件是 `\notes\settings panel\20260518_024810.png`）：
- 设置面板的左侧、右侧区域可以使用不同灰度的背景颜色来区分。
- 左侧区域的每个分类名称前面都会显示一个图标。被激活的分类有高亮样式。
- 右侧的内容使用各种卡片组合。每个卡片具有白色背景（当颜色主题为暗色时则是深灰色背景），并添加 padding、margin，使每个按钮区域或设置项都显示为独立的小卡片。这有助于设置项挨在一起时不容易区分开的问题：之前所有设置项都是直接显示文字，排列在一起时很难分辨不同设置的边界。做成小卡片之后就很容易分辨了。

### 5.4 置顶的设置如何处理

我在左侧导航区域添加了“首页”分类，它右侧的内容区域里会显示置顶的设置。

#### 被置顶的设置只有一份对应的设置元素

当左侧导航区域激活了“首页”分类时，查找所有被置顶的设置，把它们的元素从其原本所属的分类区域里移动到“首页”的置顶设置区域里。

当左侧导航区域激活了非“首页”分类时，则查找“首页”分类里是否有该分类下的设置，如果有则把这个设置从“首页”分类区域里移动到当前分类区域里。这样，用户在其他分类里依然能看到被置顶的设置，不会产生“这个设置怎么不见了”的困惑。

这样做的目的是为了让置顶的设置只有一份元素，而非多份。原因是下载器在设计时没有考虑到一个设置会存在多份的情况。如果一个设置存在多份，会带来问题：
- 由于设置项都包裹在一个 form 表单里，并且使用 form.key 的方式来读取、设置对应的值。如果被置顶的设置存在多份，那么同一个 key 会出现多次，造成问题。
- 一些模块需要向特定的设置里添加子元素，如果有多个相同的设置，就需要遍历多个设置来设置数据、创建子元素，会比较麻烦。

### 5.9 “显示高级设置”开关改为“展开/折叠所有二级分组”

因为现在每个一级导航（分类）里对设置项进行了二级分组，每个分组里的设置数量不太多，因此没必要默认隐藏部分设置了。

新的方案是：

- **不隐藏设置项本身**
- 只控制二级分组的展开 / 折叠状态
- 标题栏右侧提供 `展开/折叠全部` 按钮
- 这个按钮作用于所有二级分类，会跨全部分类统一展开或折叠二级分类

### 5.10 这个方案为什么最适合这个项目

因为这个项目的本质不是一个普通设置页，而是一个：

- 带运行状态的工具面板
- 带大量专业设置
- 同时要求高频操作效率很高

所以它更像：

- **轻量控制台**
- 而不是传统的“系统设置页”

用“左侧导航 + 右侧工作区”的方式，会比传统选项卡更适合这种工具属性。

## 9. 最终结论

如果只用一句话概括我的建议，就是：

> 把当前设置面板从“3 个页签里的长表单”，升级成“带仪表盘（首页）、导航分类、全局下载状态”的轻量控制台。

其中最关键的 4 个点是：

1. **一级分类稳定化**：把 5 个设置分类提升为主导航，并补充“首页”和“帮助”两个特殊分类。
2. **在仪表盘里集中显示置顶的设置和多种按钮**：把这些重要的设置和功能集合起来，而不是分散在多个选项卡里。
3. **下载状态全局化**：在左侧导航区底部添加一个简单的显示下载状态摘要的区域，默认不显示内容，当下载开始后显示：下载总进度、暂停下载与停止下载按钮。
4. **设置项卡片化**：每个 option 都有更清晰的标题、控件、子设置层次。

如果你后续要我继续推进，我建议下一步不是直接写最终代码，而是先做两件事：

1. 先把所有设置项映射到“一级导航 + 二级分组 + 置顶支持”的结构表。
2. 再基于这张结构表，画一版更接近真实界面的低保真原型。
